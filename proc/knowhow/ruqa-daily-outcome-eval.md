# RUQA Daily Outcome 직원 평가 가이드

Slack `#ai-work-daily` 채널의 직원별 일일 보고(09:30 / 14:30 / 17:30 구조)를 5축으로 채점해 순위·등급·관리 코멘트를 산출하는 재사용 평가 절차. 2026-05-12 ruqa_employee_evaluation 평가서를 토대로 재구성.

## 1. 5축 평가 기준 (배점 100)

| 항목 | 배점 | 판단 기준 |
|---|---:|---|
| 실제 산출물 | 35 | 링크·배포본·PR·문서·파일·코드 등 당일 결과물이 실제로 남았는가 |
| 본인 판단/결정 | 25 | AI 초안을 그대로 받지 않고 범위·우선순위·정책·UX·기술 결정을 스스로 했는가 |
| AI 활용 품질 | 20 | AI에게 맡긴 일이 구체적이고 검증/수정/반복이 있었는가 |
| 기록 성실성/완료율 | 15 | 09:30/14:30/17:30 구조, 실패/블로커, 다음 액션, 재사용 자산을 남겼는가 |
| 증거 접근성 | 5 | 링크/파일/커밋/PR이 제3자가 확인 가능하게 제시됐는가 |

## 2. 등급 변환 (5.0 환산)

| 등급 | 점수 (5.0 환산) | 의미 |
|---|---:|---|
| S | 4.5 이상 | 당일 산출물·판단·검증·기록이 모두 강함 |
| A | 4.0 ~ 4.49 | 실질 산출과 판단이 강하나 일부 리스크 |
| B | 3.3 ~ 3.99 | 의미 있는 산출은 있으나 증거/완료율/검증이 약함 |
| C | 2.5 ~ 3.29 | 활동은 있으나 결과물 또는 자기 판단 밀도가 낮음 |
| D | 2.49 이하 | 평가 가능한 산출 증거가 부족함 |

100점 → 5.0 환산은 `점수 / 20`.

## 3. 데이터 수집 절차

직원 1인당 다음 순서로 증거를 확인한다.

1. **Slack thread** — `#ai-work-daily` (`C0B3033BVTN`)에서 직원별 thread permalink 확보. 09:30/14:30/17:30 보고 구조 점검.
2. **첨부 파일** — Slack file search로 파일별 permalink·file ID·크기·타입 기록. 원문 다운로드 불가 시 본문 설명만 인용.
3. **외부 링크 확인**:
   - Notion 페이지 → 공개·권한 범위에서 실제 열람.
   - GitHub PR/repo → `gh repo view` 또는 `curl -sI`로 접근성 검증.
   - Vercel 배포 → `curl -sI`로 HTTP 200 확인.
   - 커밋 SHA → 본문에 명시됐으면 그대로 인용.
4. **외부 검증 불가 항목 명시**:
   - 비공개 GitHub repo.
   - `vscode-webview://` 로컬 미리보기 링크.
   - 커넥터 제한으로 원문 다운로드 불가한 파일.

## 4. 평가서 양식 (직원 1인 단위)

```markdown
### {이름}

Slack 근거: [{이름} thread]({permalink})

평점: {x.xx} / 5.0, {등급}

세부 점수:

| 항목 | 점수 |
|---|---:|
| 실제 산출물 | {n}/35 |
| 본인 판단/결정 | {n}/25 |
| AI 활용 품질 | {n}/20 |
| 기록 성실성/완료율 | {n}/15 |
| 증거 접근성 | {n}/5 |

확인한 산출물:
- {링크·파일·커밋·PR 라인별 정리}

좋았던 점:
- {판단 밀도, 자기 수정, 재사용 자산, 실패 인정 등 구체 사례}

아쉬운 점:
- {외부 검증 불가, 산출 분산, 자기 판단 부족, 증거 부족 등}

관리 코멘트:
{개인 맞춤 1~2문장 — "다음에 무엇을 바꾸면 점수가 오른다"}
```

## 5. 출력 문서 전체 구조

1. 평가 기준 (5축 + 등급표 + 한계 명시)
2. 종합 순위 표 (순위·이름·점수·등급·한 줄 평가)
3. 직원별 평가 (4절 양식 반복)
4. 평가 제외 항목 (역할 차이 — 대표/부대표/평가 대상 외)
5. 조직 차원 권고 (다음 round 가이드)

## 6. 조직 차원 권고 템플릿

평가서 마지막에 항상 붙이는 7개 권고. 매 round 결과를 반영해 갱신.

1. Slack 첨부만 올리지 말고 핵심 표·결정·스크린샷 3개는 본문에 붙이게 한다.
2. PR/commit/배포/Notion/파일 링크를 `실제 산출물` 항목에 한 줄씩 고정 포맷으로 적게 한다.
3. "AI에게 시킨 일"과 "내가 판단해 바꾼 일"을 반드시 분리한다.
4. 실패/블로커는 감점이 아닌 신뢰 항목으로 본다 — 단, 해결 여부와 다음 액션이 있어야 한다.
5. 한 thread에 프로젝트를 여러 개 섞으면 고성과자도 점수가 깎인다 — 프로젝트별 thread 분리 강제.
6. secret·DB password·provider key는 Slack 보고 금지 항목으로 명문화한다.
7. 최종 보고마다 "오늘 닫힌 것 / 내일 넘어간 것 / 검증 못 한 것" 세 줄을 추가한다.

## 7. 평가자 한계 명시

평가서 도입부에 반드시 명시한다.

- Slack 커넥터가 원문 다운로드를 제공하지 않으면 파일명·크기·타입·작성 시각·본문 요약만 확인 가능.
- Notion 공개/권한 링크는 접근 가능 범위에서만 열람.
- Vercel·GitHub URL은 `curl -sI` HTTP 상태 코드로 접근성만 검증.
- 비공개 GitHub repo, 로컬 `vscode-webview://` 링크는 외부 검증 불가로 처리.

## 8. 자주 발생하는 채점 패턴

- **고출력 + 운영 리스크**: 산출량은 최상이지만 secret 노출/미완료/샘플 혼재가 있으면 등급 한 단계 하향. (예: A → B+)
- **문서 산출만 풍부, 코드 반영 없음**: 본인 판단 점수는 높을 수 있으나 실제 산출물 항목에서 감점. 일반적으로 A 상단.
- **실행량은 많지만 thread 분산**: 프로젝트가 한 thread에 3개 이상이면 무엇이 닫힌 결과인지 판단 비용이 커져 B+로 떨어짐.
- **구현은 있지만 증거 없음**: 비공개 repo + 스크린샷 없음 + PR 링크 없음 — 등급은 B 또는 B+에서 캡.
- **판단력은 좋지만 링크 부재**: 사고 품질로 D 등급은 안 가지만 B- 이상 올리기 어려움.

## 9. 평가에서 제외할 카테고리

- **대표/부대표/임원 보고**: 직원 Daily Outcome 평가와 책임 범위가 다르므로 별도 운영 리뷰 기준으로 처리.
- **회사 외부 산출(웹사이트·IR deck 등)**: "직원 별 평가" 요청 범위 밖이면 명시적으로 분리.

평가 제외 항목은 본문 4절(또는 `## 참고: 직원 평가에서 제외한 항목`)에 한 번 정리해 누락된 게 아님을 분명히 한다.
